# SCALE RUNBOOK — заполнить сервис типами приложений (для Sonnet-сессии)

Это пошаговая инструкция для младшей модели (Sonnet), запущенной как агент в Claude Code.
Цель пользователя: **скачать максимум отзывов, добавить на главную как можно больше ТИПОВ
приложений (сегментов), и сделать всё двуязычным (EN оригинал + RU перевод).**

Ты — модель. В этом и фишка пайплайна: «дамп → Claude в другом окне → apply» здесь
сворачивается в одну сессию. Скрипт `dump-*` пишет самодостаточные промпт-файлы в
`classify/in/` или `translate/in/`; **ты сам читаешь каждый файл, сам пишешь JSON-ответ в
`classify/out/` или `translate/out/`, потом запускаешь `apply-*`.** Никакого API-ключа и
ручной вставки человеком не нужно.

Прочитай это полностью перед стартом. Потом иди по фазам сверху вниз, **по одному жанру за раз,
доводя его до деплоя**, и только потом берись за следующий.

---

## 0. Железные правила (нарушать нельзя — это экзистенциальная планка пользователя)

1. **НИКОГДА не выдумывай.** Каждое число должно прослеживаться до реального отзыва. Качество
   важнее охвата и скорости. Лучше пропустить (abstain), чем налепить ложную метку.
2. **Trigger — это дословная цитата.** В каждой метке поле `trigger` обязано встречаться в тексте
   отзыва ПОСИМВОЛЬНО. `apply-classify` сверяет это с БД и молча выбрасывает то, что не цитируется.
   - Дамп схлопывает пробелы (`\s+`→один пробел). В trigger — только одиночные пробелы.
   - Избегай апострофов и кавычек в trigger (кручёная `’` ≠ прямая `'` → mismatch). Для «don't»,
     «it's» бери под-кусок без апострофа.
   - Держи trigger КОРОТКИМ — меньше шансов на расхождение.
3. **Abstain — это правильно.** Пустой `labels: []` для отзыва, где ничего из таксономии честно не
   подходит: оффтоп, «good/ok/love it», чисто «дорого / жадные / сделайте бесплатно» без абьюзного
   биллинга, похвала с ошибочной 1★, расплывчатое «scam / worst app» без конкретной боли.
4. **Только pain даёт сигнал.** Похвалу не считаем за вывод (позитив легко накрутить). `stance`
   ставь честно: pain | praise | neutral, но публичный сайт показывает только pain.
5. **Игнорируй политику/религию/идеологию** — вне зоны.
6. **Pricing/billing-дисциплина.** Честный store-биллинг — НЕ проблема. Метку про биллинг ставь
   только при абьюзе: нельзя отменить, списали после отмены, скрытое автопродление, surprise charge,
   платный доступ к тому, что раньше было бесплатным. «Просто дорого» → abstain.
7. **EN/RU паритет.** Любая новая таксономия — метки нужны и `en`, и `ru`, одинаковые по смыслу.
8. **Береги prod-box.** Машина маленькая (2vCPU/1.9GB) и на ней крутится живой force-dynamic сайт.
   `ingest-*` (сеть + запись) — тяжёлый: не гоняй его параллельно с тяжёлой классификацией.
   `dump/apply` — лёгкие, но всё равно бей классификацию на жанры, не запускай всё сразу.

Эти правила в значительной степени защищены кодом: `apply-classify` отвергает выдуманные ключи,
неверный stance и нецитируемые триггеры. Чего код НЕ ловит — это плохую таксономию. Поэтому
качество таксономии (Фаза 2) полностью на тебе.

## 1. Где всё крутится (окружение)

- **Канонная БД — на prod, не локально.** Локальная dev-БД пустая. Все скрипты ходят в
  `file:/opt/badcomment/data/prod.db`.
- **SSH:** `ssh -o ConnectTimeout=20 -o ServerAliveInterval=5 -l artsaverin 51.250.11.225`
- **tsx-скрипты ОБЯЗАНЫ лежать внутри `/opt/badcomment`** (не в `/tmp`), иначе не резолвится
  `@prisma/client`. Репозиторий на prod развёрнут в `/opt/badcomment`.
- **Запуск любого скрипта на prod:**
  ```
  cd /opt/badcomment
  DATABASE_URL=file:/opt/badcomment/data/prod.db npx tsx scripts/<script>.ts <args>
  ```

- **Деплой = push в `main`.** GitHub Actions (CI: `npx tsc --noEmit` + `npm run lint`) гейтит
  SSH-деплой на prod. Перед пушем ВСЕГДА прогоняй `npx tsc --noEmit && npm run lint` локально.
- **Опасные действия (деплой, git push, массовый ingest) — спрашивай подтверждение у пользователя.**

## 2. Картина целиком: данные → смысл → язык → публикация

```
(1) INGEST     scrape отзывы в prod.db          scripts/ingest-categories.ts / ingest-list.ts
(2) TAXONOMY   автор таксономию жанра            src/lib/taxonomy.ts  (TAXONOMIES[slug])
(3) CLASSIFY   разметить отзывы по смыслу        dump-classify → ты размечаешь → apply-classify
(4) TRANSLATE  перевести pain-отзывы на RU       dump-translate → ты переводишь → apply-translate
(5) PUBLISH    проверить, закоммитить, задеплоить taxonomy.ts → push main
```

Жанр появляется на главной, как только у него есть таксономия: `src/lib/segmentCards.ts` фильтрует
сегменты по `getTaxonomy(slug) != null`. Сейчас публичны только 2 из 32 сегментов
(`language-learning`, `translators`). Все 32 сегмента с их `appIds` уже лежат в
`src/data/segments.json` — у большинства отзывы уже скачаны.

**Порядок строгий:** ingest → classify → translate. (translate дампит только отзывы, у которых уже
есть pain-метка текущей версии таксономии. classify дампит только отзывы, ещё не размеченные текущей
версией.)

**С чего начать:** бери крупные сегменты (много приложений/отзывов) первыми. Хорошие кандидаты по
размеру: `fitness-workout` (37), `calorie-diet` (35), `recipes-cooking` (34), `scanners-docs` (41),
`identifiers` (29), `smb-invoicing` (29), `phone-utilities` (26), `kids-learning` (24), `ai-chat`
(20), `planners-productivity` (19), `finance-personal` (15), `maps-navigation` (14). Делай ОДИН
жанр целиком до деплоя, потом следующий. Не копи полусделанные.

## 3. Фаза 1 — Скачать больше отзывов (ingest)

Цель «максимум отзывов» = (а) больше отзывов на приложение и (б) больше приложений в сегментах.
Это сетевой и тяжёлый шаг — согласуй с пользователем перед массовым запуском и не гоняй параллельно
с классификацией.

**A. Глубже по уже подключённым приложениям** — поднять лимит отзывов на листинг:
```
cd /opt/badcomment
INGEST_MAX_REVIEWS=400 DATABASE_URL=file:/opt/badcomment/data/prod.db npx tsx scripts/ingest-categories.ts
```
Переменные (`scripts/ingest-categories.ts`): `INGEST_COUNTRY` (us), `INGEST_TOP` (10 приложений на
категорию), `INGEST_MAX_REVIEWS` (100). Скрапит топ обоих сторов (google+apple) по `CATEGORIES`,
ingest-ит негативные отзывы. Пауза 800мс между приложениями.

**B. Добавить НОВЫЕ приложения в сегмент** — резюмируемый ingest из `selected.json`:
```
DATABASE_URL=file:/opt/badcomment/data/prod.db npx tsx scripts/ingest-list.ts
```
`ingest-list.ts` читает `selected.json` (формат: `{google, apple, name, developer, category,
bestRank}[]`), ходит в оба стора, пропускает приложения, скрапленные за последние `INGEST_RESCRAPE_DAYS`
(30) — можно убивать и перезапускать. Кандидатов готовят `harvest-candidates.ts` /
`select-candidates.ts` (см. их шапки). Чтобы новое приложение попало в сегмент на сайте — добавь его
product id в нужный массив `appIds` в `src/data/segments.json`.

После ingest проверь, сколько отзывов набралось: `DATABASE_URL=... npx tsx scripts/stats.ts`.

> Если пользователь хочет только «обогатить главную типами» без нового скрапинга — Фазу 1 можно
> пропустить: у большинства сегментов отзывы уже есть. Тогда сразу Фаза 2.

## 4. Фаза 2 — Авторинг таксономии жанра (экспертный шаг)

Это самое ответственное. Таксономия — закрытый список «нужд» (needs) с под-ветками (forks),
выведенный из ЧТЕНИЯ реальных жалоб этого жанра. Образцы для подражания уже есть в
`src/lib/taxonomy.ts`: `language-learning` (18 needs) и `translators` (11 needs). Открой и изучи их
стиль ПЕРЕД авторингом.

**Шаги:**

1. **Прочитай реальные отзывы жанра.** Возьми product id из `segments.json` для слага и выгрузи
   образцы. Быстрый путь — мини-скрипт-выборка прямо в БД (читать, не писать), например через
   `npx tsx -e` с prisma, либо `scripts/dump-samples.ts <id> <id> ...` (печатает con/pro сэмплы).
   Читай СОТНИ отзывов по нескольким приложениям, пока не увидишь повторяющиеся боли.

2. **Выведи needs из того, что люди реально жалуются** — не из головы. Need = устойчивая боль/работа,
   которую отзывы поднимают снова и снова. Fork = конкретная под-нить внутри неё. Смотри, как сделано
   у двух существующих жанров: 1 строка на need + вложенные forks, у каждого `key`, `en`, `ru`.

3. **Соблюдай форму типа `GenreTaxonomy`:**
   ```ts
   "<slug>": {
     slug: "<slug>",
     version: "<slug>-1",            // версия НА ЖАНР; ре-ран трогает только строки с другой версией
     en: "<Genre name, EN>",
     ru: "<Название жанра, RU>",
     needs: [
       { key: "<bare-key>", en: "<...>", ru: "<...>", forks: [
         { key: "<bare-key>.<sub>", en: "<...>", ru: "<...>" },
       ] },
       // ...
     ],
   }
   ```
   - `key` — латиницей, dotted для forks (`reliability.crashes`). Ключи неизменны после старта.
   - Метки `en`/`ru` сейчас сформулированы как ПРОБЛЕМЫ (не как «работы»): напр. «Не работает,
     глючит» / «Broken and buggy», «Биллинг-ловушки» / «Billing traps». Держи этот негативный
     фрейминг — он соответствует нарративу сайта «топ проблем / основные жалобы».
   - 10–20 needs на жанр — нормальный объём. Не плоди ради числа. Forks — только если они реально
     встречаются в отзывах.
   - EN/RU паритет: одинаковый смысл в обеих локалях.

4. **Добавь жанр в `TAXONOMIES`** в `src/lib/taxonomy.ts`. Больше нигде версию прописывать не надо —
   `taxonomyVersion(slug)`, `validKeys(slug)`, `renderTaxonomy(slug)` подхватят автоматически.

5. **Проверь сборку:** `npx tsc --noEmit && npm run lint`.

> Анти-фабрикация здесь: если жанр в основном состоит из мусорных приложений (VPN/cleaner/antivirus,
> scam-обёртки и т.п.) или отзывы не дают честных повторяющихся болей — НЕ выдумывай таксономию.
> Скажи пользователю, что жанр не тянет на честную разметку.

## 5. Фаза 3 — Классификация (разметка по смыслу)

1. **Дамп** (на prod, читает БД, пишет промпт-файлы в репо):
   ```
   cd /opt/badcomment
   DATABASE_URL=file:/opt/badcomment/data/prod.db npx tsx scripts/dump-classify.ts <slug>
   ```
   Пишет `classify/in/<slug>-001.txt …` + `classify/manifest.json`. Дампит только ещё не размеченные
   текущей версией отзывы (идемпотентно). Переменные: `CLASSIFY_PER_APP` (60), `CLASSIFY_WINDOW`
   (600), `CLASSIFY_BATCH` (40 отзывов на файл).

2. **Размечай — это делаешь ТЫ.** Для КАЖДОГО файла `classify/in/<slug>-NNN.txt`: прочитай его
   (внутри полный промпт с таксономией, правилами и отзывами), выполни задачу и запиши ответ в
   `classify/out/<slug>-NNN.json` (ИМЯ ТО ЖЕ, расширение `.json`). Строго следуй схеме из файла:
   ```json
   {"version":"<slug>-1","reviews":[
     {"id":"<review id>","labels":[
       {"key":"reliability.crashes","stance":"pain","confidence":0.9,"trigger":"keeps crashing"}
     ]}
   ]}
   ```
   - Каждый id из батча обязан встретиться РОВНО раз, даже с пустым `labels: []`.
   - Только JSON, без markdown-ограждений и прозы.
   - Соблюдай правила раздела 0: дословные короткие триггеры без апострофов, abstain при сомнении,
     pain/praise/neutral честно, политику игнорируй.
   - Суди каждый отзыв на его языке; trigger — на языке отзыва.

3. **Применяй:**
   ```
   DATABASE_URL=file:/opt/badcomment/data/prod.db npx tsx scripts/apply-classify.ts <slug>
   ```
   Валидирует и пишет `Review.needs` + `needsVersion`. Отчёт: `reviews classified / abstained`,
   `labels kept / rejected (of which unquotable) / ids not found`, и список отвергнутых неизвестных
   ключей. **Цель: 0 rejected, 0 unquotable, 0 ids-not-found.** Идемпотентно — можно перезапускать.

4. **Если есть unquotable/rejected:** почини соответствующий `classify/out/*.json` (чаще всего trigger
   не цитируется дословно — укороти/убери апостроф/проверь пробелы; либо неизвестный key — опечатка)
   и запусти `apply-classify` снова. Не оставляй ненулевые отказы.

## 6. Фаза 4 — Перевод на русский (RU-версия)

EN-сторона сайта показывает оригиналы отзывов — ничего доделывать не нужно. RU-сторона показывает
русский перевод для не-русских отзывов. Делай ПОСЛЕ классификации (дампятся только pain-размеченные).

1. **Дамп:**
   ```
   DATABASE_URL=file:/opt/badcomment/data/prod.db npx tsx scripts/dump-translate.ts <slug>
   ```
   Пишет `translate/in/<slug>-NNN.txt` + `translate/manifest.json`. Берёт отзывы с pain-меткой текущей
   версии, НЕ кириллические и ещё не переведённые текущей `TRANSLATION_VERSION`. Переменные:
   `TRANSLATE_WINDOW` (600), `TRANSLATE_BATCH` (40).

2. **Переводи — это делаешь ТЫ.** Для каждого `translate/in/<slug>-NNN.txt` запиши ответ в
   `translate/out/<slug>-NNN.json` по схеме:
   ```json
   {"version":"ru-1","reviews":[
     {"id":"<review id>","titleRu":"<ru title или null>","textRu":"<ru текст>"}
   ]}
   ```
   - Естественный, беглый русский. Сохраняй тон и регистр (грубое — грубым, злое — злым).
   - Верно по смыслу: не смягчать, не заострять, не сокращать, не цензурить, ничего не добавлять.
     Эмодзи и названия продуктов/фич — как есть. Если `title` null → `titleRu` null. Не теряй отзывы.
   - Только JSON, без ограждений. Каждый id ровно один раз.

3. **Применяй** (без аргумента-слага):
   ```
   DATABASE_URL=file:/opt/badcomment/data/prod.db npx tsx scripts/apply-translate.ts
   ```
   Пишет `textRu`/`titleRu`/`translationVersion`. Пустой `textRu` отвергается, оригинал не трогается.
   Отчёт: applied / blank skipped / ids not found.

## 9. Типичные грабли

- **`no segment/taxonomy for slug`** в dump-classify → жанр не добавлен в `TAXONOMIES` или слаг не
  совпадает с `segments.json`. Сначала Фаза 2.
- **unquotable (triggerMismatch)** → trigger не встречается дословно в тексте БД. Причины: апостроф
  (кручёный/прямой), множественные пробелы (дамп их схлопнул), перефраз/перевод, слишком длинный
  span. Чини в `classify/out/*.json`, перезапускай apply.
- **rejected unknown key** → опечатка в ключе или ключа нет в таксономии. Используй только ключи из
  `renderTaxonomy` (он печатается прямо в промпт-файле).
- **`@prisma/client` не резолвится** → ты запускаешь tsx из `/tmp`. Запускай из `/opt/badcomment`.
- **translate ничего не дампит** → нет pain-меток текущей версии (сделай сначала классификацию) или
  всё уже переведено / отзывы кириллические.
- **CI красный** → почини tsc/lint, НЕ пуши с `--no-verify`, делай НОВЫЙ коммит (не amend после
  упавшего хука).
- **prod тормозит** → не гоняй ingest параллельно с классификацией; разнеси по времени.

---

Резюме приоритета: бери по одному крупному жанру, доводи до живого деплоя с обеими локалями, не
выдумывай ни единого числа, и предпочитай abstain натяжке. Полнота сервиса набирается жанр за
жанром, честно.
